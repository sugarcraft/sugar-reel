# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** When wiring a new lib into the root `composer.json`, BOTH the `require` block (add `"sugarcraft/<slug>": "@dev"`) AND the `repositories` block (add `{"type":"path","url":"<slug>","options":{"symlink":true}}`) need entries — missing the repositories block produces `Could not find package` at install time even though the directory exists alongside. Pair the two edits in the same step.
- **[pattern]** When swapping a stub method (e.g. `Terminal::feed()` no-op → real Parser+Handler pipeline) for a working implementation, audit the test file FIRST and replace the corresponding stub-era test (`testFeedIsNoOpInPr1`, `testPrintsMultiByteUtf8AsSingleCell`, etc.) before running the suite — leaving the stub-era assertion in place produces a confusing failure that looks like the new wiring is broken ("expected 1 got 2" cell count for CJK after Width wiring, or "expected ' ' got 'R'" for Parser+Handler) when it's actually the test that's outdated. PHPUnit's diff output shows the new behavior on the `+` side, which is a hint to update the test rather than the code.
- **[pattern]** When writing tests that drive a VT parser/handler with raw byte sequences, set up buffer state with explicit CSI cursor-position moves (`\x1b[r;cHTEXT`) rather than chains of print + CR (`\x0D`) + LF (`\x0A`) + overwrite. Mental-tracing the latter is error-prone — rewriting them as `\x1b[1;1HAB\x1b[2;1HCD\x0A` (explicit row,col placement, then exactly one LF to trigger the behavior under test) makes the assertions trivially correct. Reserve C0/CR/LF chains for tests that are *specifically* about CR/LF semantics.
- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/sugarcraft/<lib> && cmd`, the NEXT Bash call still starts in `<lib>` — `git status` will then show paths with `../` prefixes (e.g. `modified: ../docs/index.html`) that look like they're outside the repo but aren't. A bare `ls candy-core/src/` from inside `candy-core` returns NO output (empty `{}` response from the tool, not a `No such file or directory` error) because the path is relative to the wrong cwd. Always use absolute paths or prefix every Bash call with `cd /home/sites/sugarcraft && ...` to anchor it; if you get a mysterious empty response from `ls`/`cat` or a `cat: .github/workflows/vhs.yml: No such file or directory` error on a file that obviously exists, suspect the cwd before suspecting the file. Same trap bites `php -r "file_get_contents(__DIR__.'/tests/fixtures/foo')"` — `__DIR__` in `-r` mode resolves to the cwd at exec time. Use absolute paths in `php -r` snippets.
- **[gotcha]** `gh pr view <n> --json merged` fails with `Unknown JSON field: "merged"` — that field doesn't exist. Use `--json state,mergedAt` instead: a merged PR returns `{"state":"MERGED","mergedAt":"<timestamp>"}`, an open PR returns `"mergedAt":null`. The error output helpfully lists the available fields (`additions`, `assignees`, `author`, `autoMergeRequest`, `baseRefName`, `baseRefOid`, `body`, `changedFiles`, ...) but the immediate fix is `mergedAt` not `merged`.

- **[gotcha]** `end()`, `reset()`, `next()`, `prev()`, `array_pop()`, `array_shift()`, `sort()`, etc. all take their argument by reference and CANNOT be called on a `public readonly array $foo` property — you get `Error: Cannot modify readonly property` pointing at the line that calls the function (not at any assignment), which is misleading. Use index access (`$this->foo[count($this->foo) - 1]`) or `array_key_last($this->foo)` instead.
- **[gotcha]** When a new lib subdir is added (e.g. `candy-vcr/`), the caliber pre-commit hook does NOT auto-stage untracked files in the new dir — `git add -A` would, but per project convention you stage explicit paths to avoid sweeping in `.env`-class files. Stage every new file explicitly: `git add candy-<slug>/composer.json candy-<slug>/phpunit.xml candy-<slug>/README.md candy-<slug>/CALIBER_LEARNINGS.md candy-<slug>/src/...` plus the cross-cut edits. `git status -uall <subdir>/` enumerates the per-file list when you forget what you created.
- **[gotcha]** `gh pr view <n> --json merged` fails with `Unknown JSON field: "merged"` — that field doesn't exist. Use `--json state,mergedAt` instead: a merged PR returns `{"state":"MERGED","mergedAt":"<timestamp>"}`, an open PR returns `"mergedAt":null`.
- **[gotcha]** `candy-core`'s `Program::run()` automatically dispatches three startup Msgs to `model->update()` BEFORE any caller-sent Msg, in this order: `WindowSizeMsg`, `EnvMsg`, `ColorProfileMsg`. So a test model that uses an `int $count`/`quitAfter` to issue `Cmd::quit()` after N updates needs to account for the 3 baked-in ticks. With a single `$program->send(new KeyMsg(...))` and a 2.0s loop-stop timer, a `quitAfter: 5` model only fires 4 updates (3 startup + 1 sent KeyMsg) and never reaches the quit branch — `QuitMsg should produce a quit event` then fails with no clear root cause in PHPUnit output. Use `quitAfter: 4` for that exact shape, or quit on a `KeyMsg` predicate instead of a counter when testing recorder/lifecycle plumbing.
- **[gotcha]** Streaming-writer classes whose `close()` method calls `fclose($this->fh)` (e.g. `candy-vcr`'s `Recorder::close()`) make the underlying stream resource invalid — calling `rewind($fh)` afterwards in the same test throws `TypeError: rewind(): supplied resource is not a valid stream resource`. Test pattern: do NOT call `$r->close()` if you also want to `rewind`/`fgets`/`stream_get_contents` the same `php://memory` handle to inspect the bytes. Either skip the close in tests that need to read back, or close once and then re-open via the file path (when using `Recorder::open($path)` rather than the in-memory ctor).
- **[pattern]** When PHPUnit failure messages alone don't explain a Program/Recorder/lifecycle test failure (assertion phrasing like "Failed asserting that an array contains EventKind Enum #X (Quit, 'quit')" doesn't tell you which Msgs DID flow), stage a standalone debug script at `/tmp/dbg.php` with `echo "tick $count: " . get_class($msg)` inside `update()`, run with absolute-path autoload, and inspect the printed Msg sequence. Faster than reading the 800-line Program.php dispatch. Remember to use `require_once '/home/sites/sugarcraft/<lib>/vendor/autoload.php'` (absolute), not `__DIR__ . '/vendor/autoload.php'`.
- **[gotcha]** `candy-core`'s `setupTerminal()` emits output bytes (e.g. `\x1b[?2027h` unicodeOn, alt-screen-enter, cursor-hide) BEFORE the startup `WindowSizeMsg` dispatch. Anything teeing `Program::writeOutput()` (recorder, debug tap, network mirror) sees those bytes as the FIRST event(s); the `resize` event arrives 1–3 events in. Tests asserting cassette/tap shape must scan for `Resize` rather than indexing `events[0]`. Same applies for the tail: `teardownTerminal()` emits `\x1b[?2027l` etc. AFTER `dispatch(QuitMsg)`, so the cassette would end on environmental noise unless the recorder is closed inside the `QuitMsg` branch.
- **[pattern]** Tee/sink classes attached to `Program` (recorder, debug tap, mirror) should make every `record*`/`write*` method **no-op after close()**, NOT throw `LogicException`. Reason: the runtime has `dispatch(QuitMsg) → close → teardownTerminal → writeOutput(...) → sink->record...` ordering — teardown's environmental writes (unicodeOff, alt-screen-leave, cursor-show) can't be intercepted by the caller, so a throw-on-closed sink would crash on every clean shutdown. Inverse: `Program::dispatch(QuitMsg)` must call `recorder?->recordQuit(); recorder?->close();` immediately (not at end of `run()`), so the cassette ends on a clean `quit` line and the no-op-after-close drops the teardown noise.
- **[pattern]** Cross-lib hook seams that need to live in candy-core but are implemented in a downstream lib (e.g. recorder lives in candy-vcr but the tee point is `Program::writeOutput()` in candy-core) take the form: candy-core ships an INTERFACE (`SugarCraft\Core\Recorder` with `recordResize/recordInputBytes/recordOutput/recordQuit/close`), `Program::withRecorder(?Recorder $r): self` accepts the interface, and the downstream lib provides the concrete class implementing it. Avoids candy-core depending on candy-vcr (the dependency goes the other way) and lets other consumers (debug tap, network mirror, crash bundle) slot into the same hook without pulling cassette plumbing.

- **[gotcha]** PSR-4 autoload requires one class per file with basename = class name. Putting test helper/fixture classes (e.g. `UserJsonableMsg`, `BadJsonableMsg`) at the bottom of a `FooTest.php` works ONLY because PHPUnit `require`s `FooTest.php` to instantiate the test, side-effect-loading the inline classes. If a SECOND test file (`BarTest.php`) references those fixtures and runs first in isolation (e.g. `phpunit --filter BarTest`), autoload fails — there is no `tests/.../UserJsonableMsg.php` for the autoloader to find. Move shared fixtures into their own PSR-4-conformant files (`tests/Msg/UserJsonableMsg.php`) from the start when more than one test will use them.
- **[pattern]** For a `Registry`-of-pluggable-strategies design (e.g. candy-vcr's `Msg\Registry` of `MsgSerializer`s), use unqualified class names as the routing tag for builtins (`KeyMsg`, `WindowSizeMsg` — readable cassettes/dumps) and FQCN for the catch-all that handles user types (avoids cross-namespace collision). The two coexist if the builtin's `canDecode()` checks against an explicit allowlist (`classForTag()` match expression) so it never accidentally claims a user FQCN. Catch-all serializer (`JsonableSerializer`) round-trips via `new $class(...$data)` named-arg unpacking — works when constructor parameter names match the keys returned by `jsonSerialize()`.

- **[gotcha]** `candy-core`'s `setupTerminal()` emits output bytes (e.g. `\x1b[?2027h` unicodeOn, alt-screen-enter, cursor-hide) BEFORE the startup `WindowSizeMsg` dispatch, and `teardownTerminal()` emits closing counterparts (e.g. `\x1b[?2027l`) AFTER `dispatch(QuitMsg)` once `loop->stop()` has fired. A `Recorder` (or any `writeOutput` tee) MUST stay open across teardown — closing it on `QuitMsg` drops those teardown bytes from the cassette and then a fresh `Program` on replay emits them, producing a byte-mismatch at the tail. Close the recorder at the END of `Program::run()`, not inside the `QuitMsg` branch.

Reason: the runtime has `dispatch(QuitMsg) → loop->stop → teardownTerminal → writeOutput(...) → sink->record...` ordering — teardown's environmental writes can't be intercepted by the caller, so a throw-on-closed sink would crash on every clean shutdown.
- **[gotcha]** A `candy-vcr` `Player` driving a `Program` in `SPEED_INSTANT` mode must NOT schedule every cassette event at `delay=0` via `addTimer` — React's `StreamSelectLoop` fires them all in a single tick before the program's framerate-based render tick gets a turn, so `view()` is never rendered and no output bytes are produced (`outputCount: 0` despite `outputCount > 0` in the cassette). Schedule events sequentially: dispatch one, then `addTimer($yield, $next)` with `$yield ≈ 5 ms` between events (`Player::INSTANT_YIELD_SECONDS`); in `SPEED_REALTIME` use `events[i+1]->t - events[i]->t`, clamped to `>= 0`. Same trap: replay programs need a high `framerate` (e.g. 1000.0 in `ProgramOptions`) so the renderer wakes up on the yield interval rather than the default ~60 Hz.
- **[gotcha]** A `candy-vcr` round-trip test (record session A → replay into fresh Program B → assert byte-equal) is NOT achievable with `ByteAssertion` alone — recording captures whatever the framerate-based renderer emitted during the recording loop's lifespan, the replay loop runs over a different chained-timer timeline (Player yields between events to let render fire) and emits a *cell-grid equivalent* but byte-different stream. Tests should assert `programQuitCleanly` + event-count tallies for PR4 and defer true byte-of-bytes round-trip parity to a `ScreenAssertion` (cell-grid via candy-vt) in PR5. Symptom: `byte mismatch: expected 41 bytes, got 115 bytes; first divergence at offset 32` between the recorded `tick: 6` frame and a replay that emitted `tick: 3` plus extra `\x1b[?2026h.[?2026...` synchronized-update bytes.
- **[gotcha]** `program->send($msg)` on `candy-core` `Program` enqueues the Msg directly into the dispatch queue, BYPASSING the input parser — the recorder's `recordInputBytes` hook never sees those Msgs and they don't land in the cassette as `input` events. To exercise the recorder's input path in tests, write actual bytes through the input stream resource: `$loop->futureTick(fn() => fwrite($writer, 'abc'));` where `$writer` is the write end of the `stream_socket_pair` whose read end was passed to `ProgramOptions(input: $reader, ...)`. Test models that need to keep their loop alive long enough for the recorder to capture frames should set `quitAfter: PHP_INT_MAX` and drive quit externally with `$loop->addTimer(0.020, fn() => $program->quit())`.
- **[pattern]** When a `ByteAssertion`-style diff helper produces a hex window starting at the first-divergence offset (not at offset 0), tests asserting on the diff string cannot expect the FULL hex of either expected/actual — the window only contains the divergent tail. For `("\x1b[2J", "\x1b[2K")` whose first three bytes `\x1b[2` are equal, the window starts at offset 3 and shows only `4a (J)` / `4b (K)`. Force the divergence to offset 0 (e.g. `str_repeat('A', 16)` vs `str_repeat('B', 16)`) when you want to assert the full 16-byte hex window (`4141…41` / `4242…42`).
