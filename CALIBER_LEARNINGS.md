# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** For FFI-dependent test classes (e.g. `candy-pty`'s `Pty::open()` round-trip), do NOT put the FFI/`/dev/ptmx`/PHP_OS_FAMILY skip checks in `setUp()` — that skips the entire class on FFI-less CI runners and you lose coverage on structural tests (cdef shape, env-override resolution, exception hierarchy, ioctl-constant numbers) that don't actually touch the syscalls. Instead extract a private `requirePtySyscalls()` helper and call it as the FIRST line of each test that opens a real pty. Structural tests run unconditionally; round-trip tests skip cleanly when the host can't support them.
- **[gotcha]** `.github/workflows/vhs.yml` does NOT use a YAML `matrix:` list for libs — it uses a Bash array `all=( candy-core candy-sprinkles … )` inside an `id: compute` step (around lines 51-69). AGENTS.md says "matrix lib: entry" but that's only accurate for `ci.yml`. To register a new VHS-eligible lib, edit the bash array tokens (whitespace-separated, multi-line) — `grep` for an existing entry to locate the right slot.
- **[gotcha]** `proc_close($proc)` returns `-1` (NOT the real exit code) when the child has already been reaped via a prior `proc_get_status($proc)` call — `proc_get_status` itself reaps the zombie once `running===false`, leaving `proc_close` with nothing to wait on. Pattern: in any `wait()`-style helper, poll `proc_get_status()` until `running===false`, capture `['exitcode']` FIRST, then call `proc_close()` purely to release the resource handle and discard its return value. Re-querying `proc_get_status()` after the first `running===false` returns `exitcode === -1` for the same reason — cache the value on first observation.
- **[gotcha]** PHP FFI cannot declare variadic C signatures (`int ioctl(int fd, unsigned long request, ...)`) — the cdef parser rejects `...` and `void*` variadic forms. Workaround: declare each variadic libc call with the exact concrete third-arg shape you need, e.g. `int ioctl(int fd, unsigned long request, void *arg);` in `Libc::cdef()`, and only ever pass an `FFI::new('unsigned short[N]')` (or other CData) as that arg. Multiple shapes need either separate cdef declarations under different names (not possible with libc) or a `void*` arg + manual buffer construction per call site.
- **[convention]** `TIOCSWINSZ` / `TIOCGWINSZ` ioctl numbers DIVERGE between Linux and macOS — Linux uses compact type-2 encoding (`TIOCSWINSZ=0x5414`, `TIOCGWINSZ=0x5413` from `/usr/include/asm-generic/ioctls.h`), macOS uses BSD `_IOW`/`_IOR` macro encoding that packs direction/type/group/number into the top bits (`TIOCSWINSZ=0x80087467`, `TIOCGWINSZ=0x40087468` from `/usr/include/sys/ttycom.h`). Pick at runtime via `PHP_OS_FAMILY === 'Darwin' ? DARWIN_… : LINUX_…`. The `winsize` struct itself is identical on both — four little-endian unsigned shorts `[ws_row, ws_col, ws_xpixel, ws_ypixel]`.
- **[pattern]** To verify a PTY's slave-side winsize without needing a master-fd read primitive, spawn `tput cols` / `tput lines` in the child and redirect their STDOUT to a regular file: `['/bin/sh', '-c', 'tput cols > /tmp/foo; tput lines >> /tmp/foo']`. `tput` queries `TIOCGWINSZ` on STDIN (the slave PTY), not stdout — so redirecting stdout away doesn't affect the query, and you observe what the child actually saw via plain `file_get_contents`. Set `'TERM' => 'xterm-256color'` and a sane `PATH` in the env array; without `TERM`, tput exits non-zero with an unhelpful error.
- **[gotcha]** The Bash tool's working directory persists across separate Bash calls. After `cd /home/sites/sugarcraft/<lib> && cmd`, the NEXT Bash call still starts in `<lib>` — `git status` will then show paths with `../` prefixes that look like they're outside the repo but aren't. A bare `ls candy-core/src/` from inside `candy-core` returns NO output (empty `{}` response from the tool, not a `No such file or directory` error) because the path is relative to the wrong cwd. Bare `grep candy-core/src/Cmd.php` after `cd .../candy-vcr` produces `ugrep: warning: candy-core/src/Cmd.php: No such file or directory` (and same for `grep candy-vcr/src/Player.php` after `cd /home/sites/sugarcraft/candy-vcr` — the file is at `src/Player.php` from inside that cwd; `grep .github/workflows/vhs.yml` from a lib subdir gives the same `No such file or directory` warning). Worse, a compound command like `git status --short && git add foo bar baz && git status --short` can return an EMPTY `{}` response (no stdout, no stderr, no exit-code error) when the cwd has drifted to a worktree state where the chained command finds nothing. Always anchor with `cd /home/sites/sugarcraft && ...` to start from the repo root, or pass absolute paths to every command. Same trap bites HEREDOC-written `/tmp/dbg.php` scripts: `require_once __DIR__ . '/vendor/autoload.php'` resolves `__DIR__` to `/tmp/`, NOT to the working dir of the heredoc-emitting Bash call. Use absolute autoload paths (`require_once '/home/sites/sugarcraft/<lib>/vendor/autoload.php';`) in any standalone PHP script you stage under `/tmp/`.
- **[gotcha]** PHP's `php://fd/N` stream wrapper — used to wrap a raw libc fd as a PHP stream resource via `fopen('php://fd/' . $fd, 'r+b')` — takes OWNERSHIP of the fd. Once materialised, `fclose($stream)` closes the underlying fd; calling `Libc::close($fd)` afterwards double-closes and the second close fails with EBADF (or worse, hits a recycled fd on a busy process). Pattern: in any facade that lazily materialises a stream over a fd, branch `close()` on whether the wrapper exists — `fclose($this->stream)` if yes, `Libc::close($this->fd)` only on the no-stream path. Test both branches explicitly (`testCloseRoutesThroughFcloseWhenStreamMaterialised` + `testCloseUsesLibcWhenStreamNeverMaterialised`).
- **[gotcha]** Reading from a PTY master fd via `fread()` returns `false` (NOT `''`) on Linux once every slave-side fd is closed — the kernel surfaces EIO rather than EOF on the master end. Treat `false` as logical EOF in the wrapper (return `''` to the caller); raising an exception breaks the natural "child exited cleanly" path, since at that point every consumer wants a graceful drain-then-stop, not a crash. macOS has a different but similar quirk — same handling works on both.
- **[gotcha]** A PTY's cooked-mode line discipline rewrites `\n` to `\r\n` on the master-readback path — when a child runs `echo foo` and you read the master fd, you observe `foo\r\n` (and frequently a separately-arriving `\r\n` echo from the discipline), NOT `foo\n`. Tests asserting child output must use `assertStringContainsString($marker, $captured)` or strip `\r` before comparing, never `assertSame("foo\n", $captured)`. The marker pattern (`'candy-pty-marker-' . bin2hex(random_bytes(4))`) is the canonical way to find your data inside the noise.
- **[gotcha]** `gh pr create` prints `Warning: 1 uncommitted change` to stderr but STILL creates the PR successfully and returns 0 — the warning is informational, not a failure. The PR will not contain that uncommitted file (it stays staged on the branch). Recovery: after `gh pr create` returns the URL, check `git status` for staged-but-not-committed files; if any belong with the change-set, follow up with a separate commit + push (the merge will pick it up since it's still on the same branch). Do NOT abort the PR or `git stash` blindly — the warning is benign for ancillary files (e.g. tooling auto-staging `CALIBER_LEARNINGS.md`).
- **[pattern]** For FFI-dependent test classes (e.g. `candy-pty`'s `Pty::open()` round-trip), do NOT put the FFI/`/dev/ptmx`/PHP_OS_FAMILY skip checks in `setUp()` — that skips the entire class on FFI-less CI runners and you lose coverage on structural tests (cdef shape, env-override resolution, exception hierarchy, ioctl-constant numbers) that don't actually touch the syscalls. Structural tests run unconditionally; round-trip tests skip cleanly when the host can't support them. Apply the same split for `pcntl`-dependent tests via a `requirePcntl()` helper that also checks `defined('SIGWINCH')` / `defined('SIGCHLD')` so the structural `pcntlReady()` test still runs on hosts missing the signals.
- **[gotcha]** `stream_select()` returns `false` when interrupted by a caught signal (EINTR) — e.g. SIGWINCH delivered mid-read by a `SignalForwarder`. Throwing on the false return aborts a perfectly legitimate timed read. Wrap `stream_select` in a deadline-based retry loop: capture `$deadline = microtime(true) + $timeout` BEFORE the loop, on `false` call `pcntl_signal_dispatch()` to drain pending handlers and `continue` with `$remaining = $deadline - microtime(true)`. Bail out only if `$remaining <= 0` (return null/timeout) or pcntl is missing (can't distinguish EINTR from a real error → throw). Recomputing the timeout each iteration prevents handler delivery from extending the wait beyond the caller's deadline.
- **[pattern]** PHP signal handlers installed via `pcntl_signal()` MUST NOT throw — exceptions out of a handler corrupt the call stack and produce obscure crashes. Wrap every user-supplied callback (size provider, reaper, etc.) inside `try { … } catch (\Throwable) { /* swallow */ }`. For handlers that touch a resource-owning object (e.g. SIGWINCH → `Pty::resize()`), short-circuit with an `isClosed()` / null-check at the top so a signal arriving between `close()` and PHP shutdown becomes a clean no-op instead of a thrown exception.
- **[pattern]** Drive deterministic signal-handler tests by passing `async: false` to the SignalForwarder, then `posix_kill(posix_getpid(), SIGWINCH)` followed by `SignalForwarder::dispatch()` (which calls `pcntl_signal_dispatch()`). Guard against handler bleed across tests with a `tearDown()` that calls `SignalForwarder::reset(SIGWINCH, SIGCHLD, …)` to restore `SIG_DFL` for every signal touched. To deliver a signal mid-`stream_select` for EINTR-retry tests, spawn a tiny `proc_open(['/bin/sh', '-c', 'sleep 0.03 && kill -WINCH <pid>'])` helper rather than `pcntl_alarm` (which sends SIGALRM, not the signal under test).

- **[gotcha]** Adding a new path-repo'd sibling dep to a lib (e.g. `"sugarcraft/candy-pty": "@dev"` + new `repositories` entry pointing at `../candy-pty`) requires `composer update sugarcraft/<new-dep>` to install it — `composer install` alone honours only the existing `composer.lock` and silently ignores the new `require` entry. The expected output looks like `Installing sugarcraft/candy-pty (dev-<branch>): Symlinking from ../candy-pty`. Skipping this step means autoload won't resolve the new namespace and tests fail with `Class not found`.
- **[gotcha]** When running `composer update` inside a lib whose path-repo'd siblings all live in the same monorepo on a feature branch, every sibling's lock-pin flips from `dev-master` to `dev-<your-branch>` (e.g. `Upgrading sugarcraft/candy-core (dev-master => dev-ai/candy-wish-pty-pr1-transport)`) — composer reads the symlinked source's HEAD, not the branch you booted from. The change is harmless (no source moved; the symlinks are the same) but shows up as a dirty `composer.lock` diff. Don't commit the lockfile churn unless it carries real package changes; `git checkout -- composer.lock` after the update is usually correct.
- **[pattern]** When splitting incremental work across multiple PRs (e.g. PR1 introduces an abstraction, PR2-PR4 land the real implementation), keep the PR1 stub functionally equivalent to existing behaviour — for the candy-wish PTY upgrade, `InProcessTransport::run()` walked the middleware stack inline like `HostSshdTransport` until PR2 wired up the actual PTY pump. - **[gotcha]** Public classes in this monorepo are `final` by project convention ("unless extension is part of the contract"). PHPUnit tests CANNOT mock or wrap them via an anonymous `new class($args) extends FooMiddleware { … }` — PHP fatals with `Class SugarCraft\…\Foo@anonymous cannot extend final class SugarCraft\…\Foo`. This swap also dodges the temptation to weaken the production type by removing `final` purely for testability.

- **[fix]** `gh pr merge <n> --merge --delete-branch` returning `GraphQL: Pull Request is not mergeable (mergePullRequest)` is usually a transient mergeability-recompute race on GitHub's side — most often triggered by a fresh push that just landed (e.g. a Caliber follow-up commit pushed seconds before merge), or a CI-required-checks computation still in flight. Wait a few seconds (or run any other unrelated command to give GitHub time) then retry the same `gh pr merge` invocation. Verify success via `gh pr view <n> --json state,mergedAt`. Do NOT confuse this with actual merge conflicts — those report `Pull Request is not mergeable: the merge commit cannot be cleanly created` instead. The plain "not mergeable" message without elaboration almost always clears on retry.
- **[gotcha]** When testing signal-driven behavior (SIGWINCH forwarding, SIGCHLD reapers, EINTR retry), do NOT use a fixed `usleep()` to wait for the signal target to be "ready" before delivering — PHP startup + `proc_open` shim + exec setup is variable (~100-200ms locally, often longer in CI), and a fixed delay races with that. Symptom: the test asserts `assertContains(100, $cols)` for the pre-signal baseline but observes only post-signal values like `[150, 150, 150, …]` because by the time the inner program's first sample landed, the signal had already been delivered and processed. Fix pattern: poll an observable side-effect (a tempfile being appended-to, a stream becoming readable) until you see the BASELINE state, deliver the signal, then poll again until you see the POST-SIGNAL state. Both polls bound by a generous deadline (~5s) so a hung supervisor doesn't time out the whole suite. Avoid the fixed-`usleep`-then-`posix_kill` shape entirely.
- **[gotcha]** Backgrounding a debug supervisor in a Bash one-liner via `(timeout N cmd &)` (subshell-wrapped + timeout-wrapped) BREAKS signal delivery — the captured `$!` ends up being the subshell pid or the timeout wrapper pid, not the inner PHP/cmd, so `kill -WINCH $!` either no-ops or kills the wrapper instead of the supervisor. The supervisor either exits before the loop body runs (no observable resize) or never receives the signal at all. Fix: drop the subshell and timeout — use direct `cmd & SUP_PID=$!; sleep 0.3; kill -WINCH $SUP_PID; wait $SUP_PID 2>/dev/null` so `$!` is the actual supervisor pid. If you need a timeout, build it into the inner cmd itself (`for i in 1..20; do … sleep 0.05; done` self-bounds at ~1s) rather than wrapping with `timeout(1)`.
